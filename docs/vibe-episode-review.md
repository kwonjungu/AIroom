# 1. 구동 문제 진단

## 1-1. 증상 재현 경로

학생 입장 흐름은 다음과 같이 강제됩니다.

```
타이틀 화면 → 모드 카드 클릭 → enterMode(mode)          (vibecoding.html:4098)
  └ vibeStudent 없으면 → openStudentModal()             (vibecoding.html:4100)
      └ 학교 검색 필수 → fetch('/api/school-search')     (vibecoding.html:3884)
          └ 서버: requireAuth 필수                        (server.js:644)
```

그리고 스튜디오 모드의 핵심 창작 경로:

```
칩 클릭("사과 받기 게임 만들어 줘") → sendChat()          (vibecoding.html:2778)
  └ fetch('/api/ai/chat', X-Auth-Token: AUTH_TOKEN)      (vibecoding.html:2793)
      └ 서버: requireAuth 필수                            (server.js:1848)
          └ Groq API 키 필요                              (server.js:1791~1794)
```

## 1-2. 원인 후보 (확실 → 추정 순)

### 원인 ① [확실] 인증 토큰 부재 시 첫 화면에서 완전 차단 — "전혀 안 된다"의 정체

- `vibecoding.html:788` — `const AUTH_TOKEN = localStorage.getItem('airoom_auth_token') || '';`
  이 페이지는 **독립 페이지**라서 자체 로그인 UI가 없고, 토큰은 **같은 브라우저에서 메인 백암이 SPA(`/`)에 로그인한 적이 있어야만** 존재합니다.
- 학생용 필수 API가 전부 `requireAuth` 뒤에 있습니다:
  - `/api/school-search` → `server.js:644`
  - `/api/ai/chat` → `server.js:1848`
  - `/api/vibe-progress` (GET/PATCH) → `server.js:449, 622`
- 토큰이 없으면 학교 검색이 401을 받는데, `fetchSchools`가 **에러를 삼키고 빈 배열을 반환**(`vibecoding.html:3885~3887`)하므로 화면에는 그냥 **"검색 결과가 없어요"**(3896줄)만 뜹니다.
- 학생 식별 모달(`vibecoding.html:708~727`)에는 **건너뛰기·닫기 버튼이 없고**, `saveStudentInfo()`(3916줄)는 `schoolPicked` 없이는 진행을 막습니다(`alert('학교를 검색해서 골라 줘!')`).
- **결과: 학생 기기(로그인 이력 없는 브라우저)에서는 어떤 모드에도 입장 자체가 불가능** → "전체적으로 구동이 전혀 안 된다"와 정확히 일치.
- 세션 수명도 함정: `server.js:27` `SESSION_TTL = 24시간`. 교사 PC라도 **다음 날이면 토큰이 만료**되어 같은 증상 재발. 토큰은 페이지 로드 시 1회만 읽으므로(788줄) 다른 탭에서 재로그인해도 새로고침 전엔 반영 안 됨.

### 원인 ② [확실] 스튜디오 모드는 AI 채팅 의존도가 절대적인데, 그 채팅이 같은 인증 벽 뒤에 있음

- 스튜디오의 학습 설계(`STUDIO_STORY`, vibecoding.html:3344~3357)는 4개 만들기 카드 모두 **"~게임 만들어 줘" 칩을 눌러 AI가 뼈대 DSL을 생성**하는 것이 1단계입니다.
- 토큰이 없거나 만료면 `sendChat()`이 401을 받아 **"선생님! 로그인이 풀렸나 봐요"**(vibecoding.html:2798)만 출력 → 게임 뼈대가 영영 안 만들어짐 → "만든 게임이 실행 안 된다"로 체감.
- 블록 팔레트만으로도 게임 구성은 가능하지만(엔진 자체는 정상 — 아래 1-3), 안내 문구·칩·스토리가 전부 채팅으로 유도하므로 학생은 여기서 멈춥니다.

### 원인 ③ [추정 — 서버 환경 확인 필요] GROQ_API_KEY 미설정/모델 폐기

- `server.js:1791~1794` — 키가 하나도 없으면 400 `'Groq API 키가 설정되지 않았습니다...'` 반환. 클라이언트는 이를 **"토토가 잠깐 딴생각을 했나 봐"**(vibecoding.html:2799)로만 표시하므로 교사 입장에선 원인 파악 불가.
- 요청 모델 `llama-3.3-70b-versatile`(vibecoding.html:2795)과 폴백 목록(`server.js:1761~1766`)의 `llama3-70b-8192`, `llama3-8b-8192`는 Groq에서 **이미 폐기(decommissioned)된 구세대 모델**입니다. 폴백 로직(`isGroqQuotaError`가 `model_decommissioned`를 잡음, server.js:1777)이 있어 치명적이진 않지만, 폐기 모델 2개를 헛돌며 지연·실패 확률을 높입니다.
- 진단 방법: 브라우저에서 `/api/ai/status` (인증 불필요, server.js:1754) 호출 → `hasServerKey`가 false면 이 원인 확정.

### 원인 ④ [추정 — 학교 망 환경] Blockly CDN(unpkg) 차단

- `vibecoding.html:10~12` — Blockly를 `unpkg.com` CDN에서 로드. 학교 유해차단/방화벽이 unpkg를 막으면 앱 전체가 죽고, 폴백 화면(14~23줄)의 "인터넷이 느린가 봐! 새로고침을 눌러 줘"만 반복됩니다. 새로고침으로는 해결 안 되는 상황인데 안내가 새로고침만 유도.

### 원인 ⑤ [확실 — 스튜디오 외 부수 버그] 픽셀 모드 '만약 버튼' 블록이 실행 시 무동작

- `vibecoding.html:1419` — `case'IF':break; // simplified`. `p_if_btn`(만약 버튼 A 눌려있으면) 블록은 팔레트에 있고 DSL 파싱도 되지만 **런타임에서 아무 일도 하지 않는 no-op**입니다. "블록을 끼워도 구동이 안 된다"는 인상에 기여.

### 참고: 스튜디오 엔진 자체는 정상

- `parseStudioDSL`(2006), `runStudioGame`(1737), `execStudioBody`(1708), 충돌 판정(1749~1761), EVERY 타이머(1725), 키 입력(1775~1783), 화면 D-패드(598~603) 모두 코드상 결함 없음. `_check/studio-parse-test.js` 전체 PASS 확인.
- 사소한 불일치 1건: `STUDIO_PROMPT`(vibecoding.html:3298)는 주인공 이모지로 `😀`를 허용하지만 `st_player` 드롭다운(2116줄)에는 `😀`가 없고 `🐱`가 있음. AI가 `PLAYER 😀 ...`를 내면 `setFieldValue` 실패가 조용히 삼켜져(2275줄) 🐢로 대체됨 — 치명적이진 않으나 "말한 대로 안 됨" 체감 요인.

**스튜디오의 결정적 공백 — 승리 조건을 만들 수 없음**:
`STUDIO_PROJECTS` 3번 "보석 모으기"의 힌트(vibecoding.html:3333~3334)는 *"다 모으면 END_WIN이 나오게 규칙을 만들어 봐"* 라고 안내하지만, 현재 DSL에는 **조건 블록이 하나도 없어서**(파서 2006~2059줄, 실행기 1708~1727줄에 IF 계열 부재) `점수가 n점이면 승리` 같은 규칙을 **언어적으로 표현할 방법 자체가 없습니다**. 유일한 게임 종료 수단은 "무언가에 닿는 순간"뿐입니다. → 미션이 요구하는 것을 도구가 지원하지 못하는 설계 결함이며, "게임이 제대로 안 된다"는 불만의 교육적 실체입니다.

# 3. 에피소드 개선 제안

## 3-1. 현재 에피소드 구성 평가

- **터틀 21 / 픽셀 18 / 미로 12 미션**은 챕터 잠금(`isUnlocked`, 3391줄)·별 채점·스토리 인트로가 갖춰진 완결된 학습 경로. 순차→반복→(미로에서) 선택의 흐름이 잘 잡혀 있음.
- **스튜디오는 에피소드가 아니라 "카드 4장"**(STUDIO_PROJECTS, 3325~3338): 잠금·별·완료 저장이 전부 비활성(3367~3377줄에서 스튜디오는 빈 배열/저장 안 함)이고, 교사 현황판 합계(`TOTAL_MISSIONS=51`, 3991줄)에도 스튜디오가 빠져 있어 **학생의 게임 창작 활동이 진도에 전혀 반영되지 않음**.
- 4장의 카드가 모두 "AI에게 통째로 만들어 달라고 말하기"로 시작 → 위 1번 구동 문제와 결합하면 스튜디오는 사실상 첫 단추부터 잠긴 모드가 됨. 또한 블록만으로 완주 가능한 카드가 없어 **블록 학습 → 게임 창작의 다리가 없음**.
- 보석 모으기(#3)는 도구가 지원 못 하는 목표를 제시(2-2 참조) — 성취 불가능 미션.

## 3-2. 새 블록을 활용한 신규 미션/에피소드 제안 (요약)

스튜디오를 "자유 카드 4장"에서 **가이드 에피소드 5개 + 자유 창작**으로 재구성 (별·잠금 시스템 재사용):

1. **Ep.1 무대 만들기** (배치·순차) — 블록만으로 주인공+배경+아이템 3개 배치. AI 불필요 → 인증 문제와 무관하게 첫 성공 경험 보장.
2. **Ep.2 움직여!** (이벤트) — ON_KEY 4방향 연결, 화면 밖으로 안 나가게 해 보기.
3. **Ep.3 냠냠 규칙** (충돌·변수 쓰기) — ON_TOUCH + SCORE + REMOVE. `REPEAT`(신규)로 사과 5개 깔기.
4. **Ep.4 승리의 조건** (선택·변수 읽기) — 신규 `ON_SCORE n { END_WIN }` 도입. "보석 5개를 모으면 승리" — 현재 불가능한 보석 모으기를 완성형으로 승격.
5. **Ep.5 위기와 생명** (조건 심화) — `LIVES`/`AFTER` 도입. "폭탄 3번 맞으면 패배, 30초 버티면 승리".
6. **자유 스튜디오** — 기존 "나만의 게임" 카드 유지, `CHASE`·`SOUND` 등 심화 블록 개방.

각 에피소드 완료를 `vibe-progress`에 반영(스튜디오 별 저장 활성화 + TOTAL_MISSIONS 갱신)하면 교사 현황판에서도 게임 창작 진도가 보입니다.
